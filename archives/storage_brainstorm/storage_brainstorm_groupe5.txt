Decouvertes majeures:
  - 7 types d'essence (pas 5). TOLERANCE = code mort (crafter supprime). DIURNAL/NOCTURNAL
    = MagicBees seulement, aucun effet sur ControllerStats. INSOMNIA = 1 seul tier PERFECT.
  - Multiplicateur hives MULTIPLICATIF: 4x T3 = 2.5^4 = x39.06. Avec config mid-game
    typique (20 base + 4 coffres + 2 SPEED PERFECT), drain = 5468 mB/s. 2 reservoirs
    (8000 mB) vides en 2.4 secondes. Probleme d'equilibrage probable.
  - consumeHoney() utilise getChestManager().getRegisteredChestCount() au lieu de
    networkRegistry.getChestCount(). Les coffres via relay ne coutent rien en miel.
  - StorageHiveBlockEntity.getController() = meme pattern isFormed() manquant que
    Terminal (AK) et Interface (BU). Bug systemique confirme.

Note RS: L'expert RS a contribue 14 propositions formelles (RS-5.1A a RS-5.3D) avec
references RS2. 4 corrections de priorite par l'architecte (RS-5.1C deja fait,
RS-5.3C P1→P3, RS-5.3D P2→P3, RS-5.2A P2→P1). 5 propositions uniques RS (DR-DV).
INSOMNIA: RS a signale une erreur factuelle (NORMAL au lieu de PERFECT) — corrigee.

Divergences:
  - Multiplicatif vs Additif: AE2 utilise scaling lineaire (cout fixe par device),
    RS2 aussi (somme lineaire). Create utilise addition (stress capacity/impact).
    Apica est le seul avec un multiplicateur exponentiel. Les 3 experts questionnent
    ce choix de design.
  - Granularite: RS2 a des upgrades par device (chaque importer a SA vitesse).
    Apica a des essences globales (bonus pour TOUT le reseau). Design intentionnel
    mais perte de granularite (RS).
  - Reservoir: AE2 permet l'energie depuis n'importe quelle source sur le reseau.
    Apica force 2 positions fixes dans le multibloc. Question de design, pas un bug.
  - setRemoved hive (5.3C/DN): Debat RESOLU P3. onRemove() existe dans le Block +
    validateNetworkBlocks() auto-corrige en 200 ticks max. Defense en profondeur.
  - Honey buffer: RS propose un buffer interne (pattern RS2 Controller 32k FE) pour
    absorber les fluctuations. AE2/Create n'ont pas propose cet angle [DS].
  - Validation bidirectionnelle: RS propose supprimer la double validation (hive 100t +
    controller 200t) en faveur d'une validation unidirectionnelle controller→hives [DV].

5.1 HONEY ENERGY SYSTEM (StorageDeliveryManager.consumeHoney, ControllerStats)
--------------------------------------------------------------------------------
Etat actuel: consumeHoney() (lignes 115-147) draine depuis 2 HoneyReservoirs aux
positions (-1,0,0) et (1,0,0) du multibloc. Formule ControllerStats:
  (base20 + chestCost10/coffre + essenceCost10/niveau) * hiveMultiplier * (1 - efficiency/100)
Consommation CONSTANTE (pas proportionnelle a l'activite). HONEY_CONSUME_INTERVAL = 20 ticks.
HoneyReservoir: 4000mB capacite, PARTAGE entre Storage Controller et Altar.
Fichiers: StorageDeliveryManager.java, ControllerStats.java, HoneyReservoirBlockEntity.java

--- BUGS ---

[CX] consumeHoney() utilise mauvais compteur de coffres [CONFIRME PAR ARCHITECTE]
     Quoi: StorageDeliveryManager.consumeHoney() ligne 119 utilise
           `parent.getChestManager().getRegisteredChestCount()` au lieu de
           `parent.getNetworkRegistry().getChestCount()`. Meme bug que [AI] (Groupe 2)
           pour le terminal.
     Pourquoi: ChestManager ne contient PAS les coffres enregistres via relay. Le
               formule sous-estime le nombre de coffres = consommation trop basse.
               Les coffres via relay sont GRATUITS en miel. Le joueur paie moins
               que ce qu'il devrait pour son reseau reel.
     Comment: Changer en `parent.getNetworkRegistry().getChestCount()`. 1 ligne.
     Reference: [AE2] Decouverte originale du bug par l'expert AE2 (5.1A).
                Confirmee par RS et architecte dans le code source.
                Meme pattern que [AI] Groupe 2 (display vs actual).
     Priorite: P1 BUGFIX (formule incorrecte, impact gameplay)
     Effort: 1 ligne | Risque: Aucun
     Experts: AE2 (5.1A), RS (confirmation)
     Note: Bug a DEUX manifestations : affichage terminal [AI] ET consommation
           reelle [CX]. Les deux doivent etre corriges ensemble.

--- BUGS ---

[DL] getController() sans isFormed() check [BUG SYSTEMIQUE — 3e occurrence]
     Quoi: StorageHiveBlockEntity.getController() (ligne 87) ne verifie pas
           isFormed(). Meme pattern que [AK] (terminal, Groupe 2) et [BU]
           (interfaces, Groupe 4). Troisieme occurrence du meme bug.
     Pourquoi: Si le multibloc est casse mais le bloc controller existe encore, la
               hive retourne un controller invalide. updateVisualState() verifie
               deja isFormed() (ligne 126) et gere correctement le cas LINKED vs
               ACTIVE. Mais getController() est public et tout futur appelant n'aura
               pas cette protection.
     Comment: Ajouter `if (!controller.isFormed()) return null;` dans getController().
              ATTENTION REGRESSION: Ce fix casse potentiellement onRemove() qui appelle
              getController() APRES que le multibloc soit deja demonte (isFormed=false).
              Solution: ajouter getControllerRaw() (sans isFormed check) pour les
              chemins de cleanup comme onRemove(), setRemoved().
     Reference: [AE2] Pattern systematique — chaque GridNode verifie isActive().
                Meme fix que [AK] et [BU]. Bug systemique confirme.
     Priorite: P1 BUGFIX (coherence avec AK et BU — meme fix partout)
     Effort: 2 lignes + getControllerRaw() (5 lignes) | Risque: Faible si getControllerRaw()
     Experts: AE2 (5.3A), confirmee par architecte
     Note architecte: updateVisualState() gere deja le cas non-forme. Block.onRemove()
     notifie le controller via unlinkHive(). Le risque est pour les futurs appelants
     de getController() qui ne sauraient pas verifier isFormed(). Le pattern
     getControllerRaw() est la bonne solution pour eviter la regression.

[DR] GUI decomposition consommation honey
     Quoi: Le GUI du controller affiche juste le total honey. Ajouter la

     Quoi: Pas de buffer — reservoir a 0 pendant 1 tick = honeyDepleted instantanement.

              quand reservoirs insuffisants. Depleted seulement quand buffer=0.

     Priorite: P3 DESIGN
     Effort: Moyen | Risque: Faible
     Experts: RS (RS-5.1F)

[DT] Cout non-lineaire par tier d'essence
     Quoi: Actuellement cout = level * 10 mB/s (lineaire). PERFECT ne coute que

     Pourquoi: RS Speed Upgrade a un scaling lineaire mais cumulatif. Apica

     Effort: Faible (1 methode) | Risque: Moyen (balance)

================================================================================
3. GROUPE 3 — TRANSPORT ET LIVRAISON
================================================================================

FEATURES ANALYSEES:
  3.1 Delivery System (StorageDeliveryManager ~608L)
  3.2 Delivery Bee Spawner (DeliveryBeeSpawner ~297L)
  3.3 Request Manager (RequestManager ~481L)
  3.4 Delivery Task (DeliveryTask ~328L)

Convergences Groupe 3:
  UNANIMITE 3/3: PriorityQueue au lieu de LinkedList+sort
  UNANIMITE 3/3: Map<UUID,Bee> au lieu de AABB scan O(N)
  UNANIMITE 3/3: Bug completedTaskIds eviction unsafe
  UNANIMITE 3/3: Bug item loss/dupe au reload (ASSIGNED->PENDING + bees tuees)
  AE2+RS: Retry infini sans limite
  AE2+CREATE: Extraction simulate/validate avant spawn (TOCTOU)
  AE2+RS+CREATE: Decomposition du code (608L et 481L)

Divergence notable:
  CREATE propose de simplifier radicalement (supprimer dependency tracking, FSM
  3 phases au lieu de 6). AE2 et RS veulent garder la complexite actuelle mais
  la fiabiliser. L'architecte note que le dependency tracking est utilise par
  le Crafter System (resolution recursive) — le supprimer casserait le craft
  automatique. La simplification FSM est debatable (les 6 phases gere des cas
  reels: preload, waypoints intermediaires, retry).

3.1 DELIVERY SYSTEM (StorageDeliveryManager ~608L)
--------------------------------------------------------------------------------
Etat actuel: Gestion de la queue de livraison, spawn de bees, honey consumption,
dependency tracking, splitting de taches. 608 lignes, fichier le plus complexe
du storage system. Delegue au DeliveryBeeSpawner pour le spawn effectif.
Fichiers: StorageDeliveryManager.java, DeliveryBeeEntity.java, DeliveryPhaseGoal.java

--- BUGS ---

[AR] consumeHoney() mauvais chest count [BUG]
     Quoi: StorageDeliveryManager.consumeHoney() utilise
           getChestManager().getRegisteredChestCount() au lieu de
           getNetworkRegistry().getChestCount(). Meme bug que [AI] dans le terminal.
     Pourquoi: La formule honey sous-estime le nombre de coffres du reseau
               (ignore ceux enregistres via relays). Le joueur paie moins que prevu.
     Comment: Remplacer par getNetworkRegistry().getChestCount(). 1 ligne.
     Reference: [RS] RS-expert a decouvert le pattern dans le terminal (AI).
     Priorite: P0 BUGFIX
     Effort: 1 ligne | Risque: Aucun
     Experts: RS

[AS] ASSIGNED sauvegarde comme PENDING = item dupe au reload [BUG CRITIQUE]
     Quoi: Quand le serveur sauvegarde, les DeliveryTask en etat ASSIGNED (items
           deja extraits du coffre source, en attente de spawn bee) sont serialisees
           comme PENDING. Au reload :
           1. La tache redevient PENDING → les items sont re-extraits du coffre
           2. Les items deja donnes a la bee en vol sont perdus (entite detruite)
           = Duplication des items extraits + perte des items en vol.
     Pourquoi: Le save/load ne preserve pas l'etat ASSIGNED ni les items preloaded.
               C'est un bug de data integrity critique.
     Comment: Sauvegarder l'etat exact de chaque tache (ASSIGNED, FLYING, etc.)
              et les items preloaded dans le NBT. Au reload:
              - ASSIGNED sans bee → remettre items dans le coffre source
              - FLYING → remettre items dans le coffre source (bee perdue)
              Pattern AE2: CraftingCpuLogic.writeToNBT() sauvegarde l'etat exact.
     Reference: [AE2] AE2 CraftingCpuLogic persiste l'etat exact des crafts en cours.
                Source: https://deepwiki.com/AppliedEnergistics/Applied-Energistics-2
                [RS] RS2 persiste l'etat exact des operations en cours.
                Source: https://deepwiki.com/refinedmods/refinedstorage2
                [CREATE] Create LogisticsManager est stateless (pas de persistence
                des operations en cours — les colis en transit sont perdus au reload
                mais les items source ne sont pas dupliqu's).
     Priorite: P0 BUGFIX CRITIQUE (data corruption)
     Effort: Moyen (refactoring serialization) | Risque: Moyen
     Experts: AE2, RS, CREATE (UNANIMITE 3/3)
     Note: Ce bug est le plus critique du storage system. Il peut causer
           de la duplication d'items en jeu, ce qui casse l'economie.

[AT] completedTaskIds eviction unsafe [BUG]
     Quoi: La liste completedTaskIds utilise une eviction FIFO a 100 entrees.
           Quand la liste depasse 100, les plus anciens IDs sont supprimes.
           Si une autre tache reference encore un ID evince (via dependency),
           elle ne detectera jamais que sa dependance est terminee.
     Pourquoi: La dependance reste en etat BLOCKED indefiniment, la tache n'est
               jamais executee. Les items restent bloques dans le coffre source.
     Comment: Au lieu de FIFO aveugle, compter les references. Evincer
              uniquement les IDs qui n'ont plus de dependants actifs.
              Alternative: TTL-based (evincer apres 600 ticks = 30 sec).
     Reference: [AE2] AE2 CraftingCPU ne fait pas d'eviction — les crafts termines
                sont nettoyes quand TOUS les dependants sont resolus.
                [RS] RS2 n'utilise pas d'eviction FIFO aveugle.
                [CREATE] Create LogisticsManager n'a pas de dependency tracking.
     Priorite: P1 BUGFIX
     Effort: Moyen | Risque: Faible
     Experts: AE2, RS, CREATE (UNANIMITE 3/3)

[AU] Cancel en vol sans restitution items [BUG]
     Quoi: Quand une tache est annulee alors que la bee est en vol (etat FLYING),
           les items portes par la bee ne sont pas restitues au coffre source.
     Pourquoi: La bee est killed mais ses items sont perdus dans le void.
     Comment: Dans cancelTask(), si etat == FLYING :
              1. Recuperer les items de la bee (getBeeCarriedItem())
              2. Tenter de les reinjecter dans le coffre source
              3. Si echec, drop les items au sol a la position de la bee
     Reference: [AE2] AE2 annule un craft en restituant les materiaux au reseau.
     Priorite: P1 BUGFIX
     Effort: Moyen | Risque: Faible
     Experts: AE2

[AV] Retry infini sans limite [BUG]
     Quoi: Quand onTaskFailed() est appele, la tache est remise en PENDING sans
           aucune limite de retry. Une tache qui echoue systematiquement (coffre
           plein, chemin bloque) boucle indefiniment.
     Pourquoi: Gaspillage de bees, honey, et CPU. Le joueur ne recoit aucun
               feedback que quelque chose est bloque.
     Comment: Ajouter un compteur retryCount dans DeliveryTask. Max 3-5 retries
              avec backoff exponentiel (delai croissant entre retries).
              Apres max retries: marquer FAILED, notifier le joueur via particules
              ou message dans le terminal.
     Reference: [AE2] ICraftingSubmitResult retourne des error codes (NO_CPU,
                INCOMPLETE_PLAN, etc.) avec feedback au joueur.
                Source: https://deepwiki.com/AppliedEnergistics/Applied-Energistics-2
                [RS] RS2 a des error codes et des limites de retry.
     Priorite: P1 BUGFIX
     Effort: Faible | Risque: Aucun
     Experts: AE2, RS (convergence)

[AW] Timeout FLYING manquant [BUG]
     Quoi: Une bee en etat FLYING n'a pas de timeout. Si elle est bloquee
           (pathfinding impossible, chunk unloaded, entite bugee), elle reste
           en vol indefiniment = "bee zombie".
     Pourquoi: La tache reste FLYING pour toujours, le slot de bee est occupe,
               les items sont perdus.
     Comment: Ajouter un champ flyingStartTick dans DeliveryTask. Si
              gameTime - flyingStartTick > MAX_FLYING_TICKS (600 = 30 sec),
              forcer un recall ou un fail avec restitution d'items.
     Reference: [AE2] AE2 a des timeouts sur les operations reseau.
     Priorite: P0 BUGFIX (previent perte d'items et bee zombie)
     Effort: Faible | Risque: Aucun
     Experts: AE2

[BD] Map<UUID,Bee> au lieu de AABB scan [UNANIMITE 3/3]
     Quoi: findBeeForTask() utilise level.getEntitiesOfClass(DeliveryBeeEntity,

     Reference: [AE2] AE2 utilise des registres O(1) pour tout.

[BE] TOCTOU simulate/extract avant spawn
     Quoi: La validation des items disponibles et le spawn de la bee ne sont pas
           atomiques. Entre la verification et le spawn, un autre systeme peut
           extraire les items (hopper, autre tache, joueur).
     Pourquoi: Time-Of-Check-Time-Of-Use = items extraits mais plus disponibles,
               ou items deja pris par un autre acteur.
     Comment: Pattern "simulate then commit" :
              1. simulateExtract() : verifier sans modifier
              2. spawn la bee
              3. extract() : retirer les items effectivement
              Si extract echoue apres spawn, recall la bee immediatement.
     Reference: [AE2] AE2 SIMULATE/MODULATE pattern dans toutes les operations.
                Source: https://deepwiki.com/AppliedEnergistics/Applied-Energistics-2

3.4 DELIVERY TASK (DeliveryTask ~328L)
--------------------------------------------------------------------------------
Etat actuel: 328 lignes. 6 constructeurs telescopiques. 12+ champs serialises
manuellement en NBT. FSM a 6 phases. Dependency tracking.
Fichiers: DeliveryTask.java

--- AMELIORATIONS ---

[BP] On-demand splitting
     Quoi: Au lieu de pre-splitter les taches en sous-taches au moment de la
           creation, splitter au fur et a mesure de l'execution (lazy splitting).
     Pourquoi: Le pre-splitting cree N taches immediatement meme si le reseau
               ne peut traiter que 2 en parallele. Gaspillage de memoire et
               complexite du dependency tracking.
     Comment: Une tache garde son count total. A chaque cycle, elle extrait
              min(64, remaining) et spawne une bee. La tache se termine quand
              remaining == 0.
     Reference: [AE2] AE2 CraftingCPU decompose les etapes au fur et a mesure.
     Priorite: P3 OPTIM
     Effort: Moyen | Risque: Moyen (changement de modele)
     Experts: AE2

[BQ] DataComponents + Codec au lieu de NBT manuel
     Quoi: Les 12+ champs de DeliveryTask sont serialises manuellement avec
           tag.putString/tag.getInt. Utiliser un Codec et DataComponents.
     Pourquoi: Le NBT manuel est error-prone (typos dans les cles, oublis de
               champs, pas de validation). Un Codec est type-safe et genere
               automatiquement la serialisation.
     Comment: Definir un Codec<DeliveryTask> avec RecordCodecBuilder.
              Ref: NeoForge 1.21.1 DataComponents system.
     Reference: [CREATE] AllDataComponents.java utilise des Codecs partout.
                Source: Create AllDataComponents.java
     Priorite: P2 REFACT
     Effort: Moyen | Risque: Faible
     Experts: CREATE

[BR] Simplifier FSM 6 → 3 phases
     Quoi: Le FSM actuel a 6 phases (IDLE, PICKUP, FLY_TO_WAYPOINT, FLY_TO_TARGET,
           DELIVER, RETURN). Simplifier a 3 (PICKUP, FLY, DELIVER).
     Pourquoi: Moins d'etats = moins de bugs, code plus simple.
     Comment: Fusionner FLY_TO_WAYPOINT + FLY_TO_TARGET en un seul FLY avec
              une liste de waypoints. Fusionner IDLE + RETURN en un seul etat.
     Reference: [CREATE] FrogportBlockEntity utilise seulement 2 etats.
                Source: Create FrogportBlockEntity.java
     Priorite: P3 REFACT (decision gameplay — les waypoints sont visuellement
               importants pour le joueur)
     Effort: Eleve | Risque: Eleve (regression potentielle)
     Experts: CREATE
     Note architecte: Les 6 phases grent des cas reels (waypoints intermediaires,
     preload de l'item avant vol, retour a la hive). Simplifier risque de perdre
     des fonctionnalites. A evaluer avec prudence.

[BS] Builder pattern pour DeliveryTask
     Quoi: Les 6 constructeurs telescopiques sont remplacees par un Builder.
     Pourquoi: 6 constructeurs avec des combinaisons de parametres differentes
               sont difficiles a lire et a maintenir.
     Comment: DeliveryTask.builder().from(pos).to(pos).item(stack).priority(p).build()
     Reference: [RS] RS-expert. Pattern classique Java.
     Priorite: P2 REFACT
     Effort: Faible | Risque: Aucun
     Experts: RS

[BT] cancelDependentTasks() iteratif au lieu de recursif
     Quoi: cancelDependentTasks() est recursif sans limite de profondeur.
           Une chaine longue de dependances peut causer un StackOverflow.
     Pourquoi: Le Crafter System cree des chaines de dependances recursives
               (max profondeur 8). 8 niveaux ne causent pas de StackOverflow,
               mais c'est une bombe a retardement si la limite change.
     Comment: Remplacer la recursion par une boucle iterative avec une
              Stack/Queue explicite.
     Reference: [RS] RS-expert. Pattern classique d'elimination de recursion.
     Priorite: P2 BUGFIX PREVENTIF
     Effort: Faible | Risque: Aucun
     Experts: RS

1.3 NETWORK VALIDATION
--------------------------------------------------------------------------------
Etat actuel: Polling toutes les 200 ticks (10s). Guards hasChunkAt() partout.
Fichiers: StorageControllerBlockEntity.java (validateNetworkBlocks),
          StorageEvents.java, RequestManager.java, DeliveryBeeSpawner.java

--- AMELIORATIONS ---

[L] Event-driven validation (etendre onBlockBreak)
    Quoi: StorageEvents.onBlockBreak() existe DEJA mais ne se declenche que
          quand le joueur est en edit mode (check StorageEditModeHandler.isEditing).
          Retirer/relaxer le gate edit-mode pour le path de cleanup.
    Pourquoi: Un coffre/terminal/interface detruit hors edit mode n'est detecte
              qu'apres 10 secondes (200 ticks). Avec l'event, c'est immediat.
    Comment: Retirer le check isEditing() a la ligne 377 de StorageEvents.java
             pour le chemin de cleanup (pas pour le chemin d'enregistrement).
    Reference: [AE2] AE2 utilise addNode/removeNode event-driven.
               Source: https://deepwiki.com/AppliedEnergistics/Applied-Energistics-2/1-overview
               [RS] RS2 utilise BlockEvent.BreakEvent.
               Source: https://deepwiki.com/refinedmods/refinedstorage2/3.2-neoforge-platform
    Priorite: P0 QUICK WIN (1 ligne de code, infrastructure existe)
    Effort: 1 ligne | Risque: Aucun
    Experts: AE2, RS (convergence forte)
    Note: Decouverte de l'architecte — le code existe, il est juste gate.

[M] Amortized validation (round-robin)
    Quoi: Au lieu de valider TOUS les blocs en une fois toutes les 200 ticks,
          valider N blocs par tick en rotation.
    Pourquoi: Un reseau avec 200 coffres = 200 checks en 1 tick = lag spike.
              Amortir = lisser la charge.
    Comment: Index rotatif, valider ceil(totalBlocks / SPREAD_TICKS) par tick.
    Reference: [RS] RS propose cette approche.
               [AE2] AE2 TickManagerService distribue les ticks entre composants.
    Priorite: P2
    Effort: Moyen | Risque: Faible
    Experts: RS, AE2 (complementaire)

[N] TTL keepAlive (heartbeat)
    Quoi: Chaque bloc reseau envoie un "heartbeat" au controller periodiquement.
          Si pas de heartbeat pendant N ticks, le bloc est considere invalide.
    Pourquoi: Inverse le modele: au lieu du controller qui poll, les blocs
              signalent leur presence.
    Comment: Pattern LogisticallyLinkedBehaviour de Create.
    Reference: [CREATE] Create LogisticallyLinkedBehaviour utilise un TTL cache.
               Source: Create LogisticallyLinkedBehaviour.java
    Priorite: P3 (alternative a L+M, pas cumulable)
    Effort: Moyen | Risque: Moyen
    Experts: CREATE
    Note: Mutuellement exclusif avec L+M. Choisir l'un ou l'autre.

1.4 STORAGE CHESTS
--------------------------------------------------------------------------------
Etat actuel: 3 types vanilla (Chest, Trapped Chest, Barrel). Flood fill BFS
sans limite. Edit mode pour enregistrement.
Fichiers: StorageChestManager.java (~169L), StorageHelper.java

--- AMELIORATIONS ---

[P] Detection par IItemHandler capability
    Quoi: Remplacer la whitelist 3 blocs (StorageHelper.isStorageContainer) par
          une detection via la capability IItemHandler de NeoForge.
    Pourquoi: Supporterait automatiquement tous les conteneurs moddeds
              (Iron Chests, Sophisticated Backpacks, Drawers, etc.)
    Comment: Remplacer `block instanceof ChestBlock || block instanceof BarrelBlock`
             par `level.getCapability(Capabilities.ItemHandler.BLOCK, pos, dir)`.
    Reference: [AE2] Le Storage Bus AE2 supporte tout IItemHandler.
               Source: https://guide.appliedenergistics.org/1.20.1/ae2-mechanics/import-export-storage
               [RS] RS2 External Storage supporte tout IItemHandler.
               Source: https://deepwiki.com/refinedmods/refinedstorage2/2.2-storage-system
               [CREATE] CapManipulationBehaviourBase detecte par capability.
               Source: Create CapManipulationBehaviourBase.java
    Priorite: P2
    Effort: Moyen (necessite aussi migration Container -> IItemHandler dans
            StorageItemAggregator et ContainerHelper) | Risque: Moyen
    Experts: AE2, RS, CREATE (UNANIMITE 3/3)
    Note: Implique un refactoring de StorageItemAggregator qui utilise
          actuellement `Container` (vanilla) pour toutes les operations.
          4+ fichiers impactes: StorageItemAggregator, ContainerHelper,
          StorageChestManager, DeliveryContainerOps.

[Q] Flood fill cap (~64 coffres)
    Quoi: Limiter le nombre de coffres enregistrables par flood fill BFS.
    Pourquoi: Un BFS sans limite dans une zone dense = potentiel lag spike.
              Un joueur pourrait enregistrer des centaines de coffres d'un clic.
    Comment: Ajouter `if (registered.size() >= MAX_CHESTS) break;` dans le BFS.
    Reference: [AE2] AE2 limite les channels (8 normal, 32 dense) ce qui
               contraint indirectement le nombre de storage.
               [RS] RS n'a pas de limite explicite mais la performance degrade.
    Priorite: P0 QUICK WIN
    Effort: 5 lignes | Risque: Aucun
    Experts: AE2, RS (convergence)

--- OPTIMISATIONS ---

[R] VersionedInventoryWrapper sur les coffres
    Quoi: Wrapper les coffres du reseau avec un tracker de version. Si le
          contenu d'un coffre n'a pas change, skip le recalcul d'agregation.
    Pourquoi: refreshAggregatedItems() itere TOUS les coffres TOUS les slots
              a chaque refresh. Si 90% des coffres n'ont pas change, c'est
              90% de travail inutile.
    Comment: Pattern VersionedInventoryWrapper de Create.
             Chaque coffre a un compteur de version incremente a chaque
             modification. Le refresh ne recalcule que les coffres modifies.
    Reference: [CREATE] Create VersionedInventoryWrapper +
               VersionedInventoryTrackerBehaviour.
               Source: Create VersionedInventoryWrapper.java,
               VersionedInventoryTrackerBehaviour.java
    Priorite: P2
    Effort: Moyen | Risque: Faible
    Experts: CREATE (proposition unique)
    Note: Necessite de wrapper les Container/IItemHandler des coffres.
          Complexifie la gestion mais gros gain de performance potentiel
          pour les reseaux avec 50+ coffres.
